The Agentic Turn

How AI agents are rewriting software architecture — a new architectural style, a new middleware, and a quality attribute your textbook doesn't have.

Duration ~45–50 min Format Seminar + open discussion Date 15 Jul 2026
The premise

Your newest component
is an unreliable actor.

Classical architecture rests on one assumption: components are deterministic. Same input, same output. Every quality attribute you've learned — reliability, testability, modifiability — is built on it.

An LLM agent behaves less like a library and more like an untrusted, fallible actor. That single break in the assumption is what makes this an architecture problem, not a coding one.

Ask the class

In a system you've built, where would non-determinism scare you most — the login check? a payment? a database write?

01 · A new architectural style

The design patterns of the AI era.

Just as GoF gave object-orientation a shared vocabulary, agentic systems now have recurring, named structures. These are architectural patterns — they shape control flow and component boundaries, not just code.

◆ Reflection ◆ ReAct ◆ Plan-and-Execute ◆ Tool Use ◆ Multi-Agent Collaboration ◆ Memory Management ◆ Human-in-the-Loop
  • Context engineering has replaced prompt engineering — you architect what the agent sees on every call.
  • Memory is now a first-class primitive, structured into short-term / working / long-term tiers.
Ask the class

Each of these echoes something you already know. Which classic idea does Reflection remind you of? (Hint: reviewing your own pull request.)

01 · Patterns in action

Each pattern, made concrete.

PatternA concrete example
ReflectionA coding agent writes a function, re-reads it, spots an off-by-one bug, and fixes it before showing you — self-review.
ReAct"Is /login down?" → think → curl the endpoint → see 500 → think → read the logs → answer.
Plan-and-Execute"Migrate this project to TypeScript" → plan the steps → rename, add types, fix errors, run the build.
Tool UseInstead of guessing a total, the agent runs a real SQL query or calls a calculator API and uses the result.
Multi-AgentA planner + coder + reviewer agent split one task — like decomposing a system into services.
MemoryA tutoring agent recalls which topics a student struggled with last week and adapts today's session.
Human-in-the-LoopThe agent drafts a DB migration but waits for your click before running it on production.

02 · A new middleware

MCP — the SOA moment for agents.

The Model Context Protocol standardizes how an agent connects to tools, data and services — the same problem SOA and middleware solved, now for AI.

  • Example: instead of custom code per integration, point one agent at MCP servers for GitHub, Google Drive, Slack, and Postgres — plug-and-play.
  • Created by Anthropic, donated to the Linux Foundation (Dec 2025).
  • Adopted by every major provider — Anthropic, OpenAI, Google, Microsoft, Amazon.
  • Roadmap: servers that act as agents themselves → "fractal" systems.

03 · Quality attributes, redefined

You can't remove non-determinism.
You bound it.

  • Bound the variance — pin the model version, seed, and temperature.
  • Test for meaning, not bytes — "Paris is the capital" vs "The capital is Paris": both correct, exact-match fails.
  • Measure statistically — 5–10 rollouts per prompt; report mean ± std-dev, like a benchmark.
  • New QAs join the list — evaluability, guardrails, cost-per-call, observability.

Example in the wild: classify the same support ticket twice → "billing" once, "account" the next. Your test suite passes 9 runs out of 10.

Why pin versions: a vendor updates the model overnight; yesterday's prompt now formats output differently and breaks your parser.

Ask the class

How would you write a test for a component that is only right 90% of the time — and what would "passing" even mean?

04 · The architect's changing role

From gatekeeper to guardrail.

Yesterday
Block change
at review gates
→
Now
Embed guardrails
so autonomy scales safely
  • Guardrail examples: the coding agent may only write inside /workspace; its DB credential is read-only; stop after $5 of API calls; deleting data needs a human click.
  • Agents need architecture they can consume programmatically — without authoritative context they optimize locally and create long-term debt.
Ask the class

Pick a system you use daily: where is a human-approval gate worth the friction — and where is it just theatre?

Putting it together · one system, every idea

Worked example: an automated PR-review agent.

1Reads the pull request & related files from GitHubMCP · Tool Use
2Reasons over the diff, pulls the tests it needs, checks each changeReAct
3A security reviewer and a style reviewer run in parallelMulti-Agent
4Re-reads its own comments, drops the false positivesReflection
5Scored nightly on a labelled eval set; model version pinnedEvaluability
6It comments — but a human still approves the mergeHuman-in-the-Loop

The durable design here isn't the agent's code — it's the boundaries: read-only by default, can't merge, always evaluated. That's the architecture.

Ask the class

If an agent already flags the obvious bugs, what should a human code review now actually check?

The one thing to remember
AI didn't make architecture obsolete. It made it the only durable layer.

When code is cheap to regenerate and non-deterministic at runtime, the architecture — the boundaries, constraints and quality attributes — is what holds.

Ask first

Put the question to them before you read the line above: if an agent can regenerate the code cheaply, what becomes the durable part of the system?

Recap · the six beats

What to carry out of the room.

  • The old world assumed deterministic components. Agents break that.
  • Agents bring their own patterns — the GoF of AI.
  • They connect through MCP — the new middleware / SOA.
  • They force new quality attributes — you bound non-determinism, not remove it.
  • The architect shifts from gatekeeper → guardrail designer.
  • Punchline: architecture is now the only durable layer.
Leave them with this

Which of your past projects would break first if one component suddenly became non-deterministic — and what would you change now?
